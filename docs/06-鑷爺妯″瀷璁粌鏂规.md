# 自研模型训练方案

> 版本：v0.1
> 日期：2026-06-23
> 范围：轨道 B 自回归 Transformer 模型的训练落地
> 算力：本地 4090 做小规模验证，租云 GPU 做正式训练

---

## 0. 定位与目标

- 把 `02-算法研究路线.md` 的算法实现，落到**可训练、可评估、可复现**的工程上
- **不追 SOTA**：目标是从零训练出能产出可接受音频的模型，理解全流程
- 两类训练：
  - **codec 训练**（第 2 层）：VQ-VAE / RVQ
  - **生成模型训练**（第 3-4 层）：自回归 Transformer + 文本条件

---

## 1. 训练全流程总览

```
数据集 → 预处理 → [训练 codec] → 用 codec 把音频编码成 token 缓存
                                        │
                                        ▼
                         [训练自回归 Transformer]（在 token 上）
                                        │
                                        ▼
                              [加文本条件继续训练]
                                        │
                                        ▼
                         评估(FAD/CLAP) → checkpoint → 接入平台
```

**关键**：codec 训好后，把整个数据集预先编码成 token 存盘，生成模型训练时直接读 token，**大幅加速**（不用每步重新编码音频）。

---

## 2. 数据预处理（训练前统一步骤）

详见 `03-数据集调研.md` 第 4 节，落成脚本：

```python
# preprocess.py 核心步骤
# 1. 扫描音频文件
# 2. ffmpeg: 转 wav、重采样到 32kHz、单声道
# 3. 切片成固定长度（如 10s）
# 4. 响度归一化 (pyloudnorm, 目标如 -14 LUFS)
# 5. 去静音 / 过滤损坏
# 6. 划分 train/val
# 输出: 规整的 wav 片段 + manifest.json（路径、时长、可选文本）
```

manifest 示例：
```json
{"audio_path": "data/clips/0001.wav", "duration": 10.0, "text": "calm piano"}
```

## 3. 阶段 A：训练 codec（第 2 层）

### 3.1 模型
- 小型 VQ-VAE → 进阶 RVQ（多码本）
- 编码器/解码器：1D 卷积栈；中间量化层

### 3.2 训练配置（4090 起步）
| 项 | 建议起点 |
|----|---------|
| 数据 | NSynth / MAESTRO 子集（几小时） |
| 采样率 | 32kHz |
| batch | 视显存，16-32 个短片段 |
| loss | 重建loss + commitment loss + codebook loss |
| 优化器 | Adam, lr 1e-4 |
| 显存 | 4090 充裕 |

### 3.3 验收
- [ ] 重建音频能听出原貌（不要求完美）
- [ ] codebook 利用率合理（无大量死码）
- [ ] 能稳定把音频 ↔ token 互转

> codec 是地基，**先在 4090 上调通**，规模小、迭代快，没必要租卡。

---

## 4. 阶段 B：训练自回归 Transformer（第 3 层）

### 4.1 准备：预编码数据
用训好的 codec 把整个数据集编码成 token，存为缓存文件（`.npy`/`.pt`），训练时直接加载。

### 4.2 模型
- Transformer decoder（参考 nanoGPT 风格自己实现）
- 先**单码本**简化版跑通 → 再上**多码本 delay pattern**（MusicGen 关键）

### 4.3 训练配置
| 项 | 4090 小规模 | 租卡正式训练 |
|----|------------|-------------|
| 模型规模 | 几十 M 参数 | 几百 M |
| 数据 | MAESTRO/FMA-small | FMA-medium+ |
| batch | 小，配合梯度累积 | 大 |
| 序列长度 | 短（如 5-10s 对应的 token） | 更长 |
| 优化器 | AdamW, warmup + cosine | 同 |
| 技巧 | 梯度检查点、混合精度(amp) | 多卡 DDP |
| 显存 | 勉强-充裕（看规模） | 多卡分摊 |

### 4.4 验收
- [ ] 训练 loss 正常下降
- [ ] 能自回归采样出 token → codec 解码 → 出音频
- [ ] 生成音频有音乐性（哪怕粗糙），非纯噪声

---

## 5. 阶段 C：文本条件训练（第 4 层）

- 数据换成 **MusicCaps**（音频-文本配对）+ MTG 标签
- 用预训练 T5/CLAP 编码文本，作为条件注入 Transformer
- 加 classifier-free guidance（训练时随机丢弃条件）
- 验收：不同 prompt 产出可区分的风格

---

## 6. 评估（第 5 层）

| 指标 | 衡量 | 工具 |
|------|------|------|
| **FAD** | 生成分布 vs 真实分布的距离（越低越好） | frechet-audio-distance 库 |
| **CLAP score** | 生成音频与文本 prompt 匹配度 | CLAP 模型 |
| 重建 SI-SNR | codec 重建质量 | 自实现 |
| 人工 MOS | 主观听感 | 自己/小范围打分 |

建立标准评估流程：固定一组 prompt + 验证集，每个 checkpoint 跑同一套，横向对比。

---

## 7. 租卡训练操作

### 7.1 何时租卡
- codec 训练：**不用租**，4090 够
- 生成模型小规模验证：4090
- 生成模型正式放大（更大模型/更多数据/更长序列）：**租卡**

### 7.2 平台选择（见 `01` 第 3.2 节）
- 实验/微调：Vast.ai / RunPod / AutoDL（按小时，便宜灵活）
- 多卡多天：Lambda / 云厂商 spot

### 7.3 租卡工作流
```
1. 本地把代码 + 预处理好的 token 缓存打包（或放对象存储/Git LFS）
2. 选实例（如 1-4× A100/H100），用带 CUDA 的镜像
3. 拉代码、装依赖（建议 Docker 镜像固化环境，避免每次重配）
4. 挂载/下载数据 → 启动训练（开 wandb/tensorboard 远程监控）
5. 定期把 checkpoint 同步回对象存储（防实例回收丢失）
6. 训练完，下载最终权重 → 释放实例（停止计费！）
```

### 7.4 省钱要点
- 用 spot/可抢占实例（便宜，但要做好 checkpoint 续训）
- 数据和环境预先固化成镜像，减少计费时间内的配置耗时
- 小规模先在 4090 验证代码正确，**别拿租来的卡 debug**

---

## 8. Checkpoint 与复现管理

- **checkpoint**：定期保存（含 optimizer state，支持续训）；命名带 step/epoch
- **配置**：每次训练的超参写进 config 文件随 checkpoint 存档
- **实验追踪**：用 wandb 或 tensorboard 记录 loss、样本、指标
- **可复现**：固定随机种子；记录数据版本、代码 commit、依赖版本
- **权重产物**：训练产出最终权重 → 拷回平台 `inference/` 供 `custom.py` 引擎加载

## 9. 训练常见问题

| 问题 | 可能原因 | 应对 |
|------|---------|------|
| codec 重建糊/噪 | 模型太小 / 训练不足 / 码本坍缩 | 加大模型、调 commitment loss、重启死码 |
| codebook 死码多 | 学习率/初始化 | EMA 更新码本、码本重置 |
| Transformer loss 不降 | lr 不当 / 数据问题 | warmup、检查 token 缓存正确性 |
| 生成全是噪声 | 采样/解码 bug / 训练不足 | 先验证 teacher-forcing 重建链路 |
| 生成重复单调 | 过拟合 / 采样温度低 | 加数据、调 temperature/top-p |
| 显存 OOM | 模型/序列/batch 太大 | 梯度检查点、amp、梯度累积、租多卡 |

---

## 10. 与平台合流

训练产出的权重，通过推理层的引擎抽象接入（见 `05` 第 8 节）：
1. 实现 `engines/custom.py`，加载自研权重，实现统一 `generate` 接口
2. 在 `models` 表注册 `custom-v1`
3. 前端模型下拉即可选用，与 MusicGen 并存对比
4. 因为平台架构早已自建，接入只动推理层，零改动平台层

---

## 11. 落地顺序建议

1. 先写好 `preprocess.py`，在 NSynth 上跑通预处理
2. 4090 训 codec（阶段 A），验收重建质量
3. 预编码数据 → 4090 训单码本 Transformer（阶段 B 简化版）跑通链路
4. 上多码本 delay pattern；需要放大时租卡
5. MusicCaps 上做文本条件（阶段 C）
6. 搭评估流程，横向对比 checkpoint
7. 满意的权重接入平台
