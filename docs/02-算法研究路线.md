# 音乐生成算法研究路线（轨道 B）

> 版本：v0.1
> 日期：2026-06-23
> 生成模型路线：**自回归 Transformer**（MusicGen 路线）
> 算力：本地 RTX 4090（24GB）做开发与小规模训练，必要时租卡

---

## 0. 总原则

- **逐层往上**：每层先读懂原理 → 亲手用 PyTorch 实现一个小版本 → 验收通过再上一层
- **小步快跑**：在 4090 上用小数据集、小模型快速迭代，先跑通再放大
- **不啃全量源码**：MusicGen/AudioCraft 源码作为参考，不一开始通读
- **核心投入在第 2、3 层**：神经编解码 + 自回归生成是理解的关键

整条链路的核心思想（先建立全局认知）：

```
原始音频波形（连续、超长序列，无法直接建模）
   │  第2层：神经编解码器（VQ-VAE / RVQ）压缩
   ▼
离散音频 token 序列（像"文字"一样可被语言模型处理）
   │  第3层：自回归 Transformer 逐 token 预测
   ▼
生成的 token 序列
   │  第2层的解码器还原
   ▼
生成的音频波形
   ▲  第4层：文本条件控制整个生成过程
```

## 第 1 层：音频信号基础

**目标**：理解音频如何数字化表示，为什么不能直接生成原始波形。

### 要懂的概念
- 采样率（sample rate）、量化位深、声道
- 时域 vs 频域；STFT（短时傅里叶变换）
- 梅尔频谱（mel-spectrogram）、梅尔刻度为何更符合听觉
- **核心认知**：48kHz × 30s ≈ 144 万个采样点 → 序列太长无法直接自回归建模 → 必须先压缩（引出第 2 层）

### 要读
- torchaudio / librosa 官方教程
- 任一份"音频信号处理基础"讲义（STFT、mel 部分）

### 亲手实现（验收标准）
- [ ] 用 torchaudio 加载音频，做重采样、单声道化
- [ ] 实现 波形 → mel 频谱 → 可视化；再 频谱 → 波形（Griffin-Lim 重建）
- [ ] 能解释清楚：为什么生成任务里要先压缩音频

**预估**：3-5 天

---

## 第 2 层：神经音频编解码 ⭐（最核心）

**目标**：亲手实现把连续音频压成**离散 token**、再还原的编解码器。这是音乐生成的地基。

### 要懂的概念
- 自编码器（AutoEncoder）→ VAE → **VQ-VAE**（矢量量化）
- codebook（码本）、最近邻量化、直通估计（straight-through estimator）
- **RVQ（残差矢量量化）**：用多个码本逐级逼近，EnCodec/SoundStream 的关键
- commitment loss、codebook loss、重建 loss

### 要读（必读）
- **VQ-VAE** 论文《Neural Discrete Representation Learning》
- **SoundStream**（Google）—— 神经音频 codec 开山
- **EnCodec**（Meta）《High Fidelity Neural Audio Compression》—— MusicGen 的输入基础，重点读

### 亲手实现（验收标准）
- [ ] 实现一个最小 VQ-VAE：音频/频谱 → 离散码 → 重建
- [ ] 在小数据集上训练，重建音频能听出原貌
- [ ] 进阶：加入 RVQ，多码本提升重建质量
- [ ] 能打印出一段音频被编码成的 token 序列（为第 3 层做输入）

**为什么最该投入这里**：搞懂"音频如何变成离散 token"，第 3 层的自回归生成就水到渠成。这是整个领域的认知拐点。

**预估**：2-3 周（含训练调试）

---

## 第 3 层：自回归 Transformer 生成 ⭐

**目标**：在第 2 层产出的音频 token 上，训练一个 Transformer 像语言模型一样逐 token 生成音乐。

### 要懂的概念
- Transformer decoder、自注意力、因果掩码（causal mask）
- 自回归生成：给定前文 token 预测下一个 token
- **多码本带来的难题**：RVQ 每个时间步有多个 token（多个码本），如何建模？
  - MusicGen 的 **codebook interleaving / delay pattern**（码本延迟交错模式）—— 重点理解
- 采样策略：temperature、top-k、top-p
- teacher forcing 训练

### 要读
- 《Attention Is All You Need》（Transformer 基础，若不熟）
- **MusicGen**《Simple and Controllable Music Generation》—— 核心论文，重点看 token 建模和 delay pattern
- nanoGPT（Karpathy）—— 学一个干净的自回归 Transformer 实现范式

### 亲手实现（验收标准）
- [ ] 先在**单码本**简化设定下，训一个小 Transformer 预测音频 token
- [ ] 能自回归生成 token 序列 → 经第 2 层解码 → 还原出（粗糙的）音频
- [ ] 进阶：实现多码本的 delay pattern，建模 RVQ 的多 token
- [ ] 实现 top-k/top-p 采样，对比生成多样性

**里程碑** 🎯：走到这一步，你能用**自己实现的 codec + 自己实现的 Transformer**，在小数据集上训练并生成音频。这是质变点。

**预估**：3-4 周

---

## 第 4 层：文本条件控制

**目标**：让文本描述（prompt）控制生成什么样的音乐。

### 要懂的概念
- 条件生成：如何把文本信息注入自回归生成
- 文本编码器：T5 / **CLAP**（音频-文本对比学习，能对齐两个模态）
- cross-attention 注入条件 vs prefix 注入
- classifier-free guidance（无分类器引导，控制条件强度）

### 要读
- **CLAP** 论文（音频-文本对齐）
- MusicGen 中的 text conditioning 部分
- classifier-free guidance 原理

### 亲手实现（验收标准）
- [ ] 用预训练 T5/CLAP 编码文本，作为条件输入第 3 层的 Transformer
- [ ] 训练后，不同文本 prompt 能产生可区分的音乐风格
- [ ] 实现 classifier-free guidance，调节条件强度

**预估**：2-3 周

---

## 第 5 层：评估

**目标**：量化生成质量，指导迭代。

### 要懂的概念
- 音乐生成无简单 accuracy
- **FAD（Fréchet Audio Distance）**：生成分布与真实分布的距离
- **CLAP score**：生成音频与文本 prompt 的匹配度
- 人工主观评测（MOS）

### 亲手实现（验收标准）
- [ ] 实现 FAD 计算脚本
- [ ] 用 CLAP score 评估文本-音频匹配
- [ ] 建立一套对比不同 checkpoint 的评估流程

**预估**：1 周

## 推荐入门数据集（小规模、能在 4090 快速迭代）

| 数据集 | 内容 | 用途 | 备注 |
|--------|------|------|------|
| **NSynth** | 单音符乐器音 | 第 2 层练 codec | 小、干净、易上手 |
| **MAESTRO** | 钢琴演奏 | codec + 生成 | 单乐器，结构清晰 |
| **FMA (small)** | 免费音乐片段 | 生成 + 文本 | 注意各曲目授权 |
| **MTG-Jamendo** | 带标签音乐 | 文本条件 | 有风格/情绪标签 |
| **MusicCaps** | 音频-文本描述对 | 第 4 层文本条件 | Google 发布，配对好 |

**策略**：第 2、3 层先用 NSynth/MAESTRO 这类单乐器、小规模数据快速验证算法正确性；第 4 层再上 MusicCaps 这类带文本描述的数据。合规性详见 `03-数据集调研.md`（待写）。
